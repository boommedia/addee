Critical Issues (fix before next deploy)

C1 — Title template duplication: every page renders "Bloggy | Bloggy" or "— Bloggy | Bloggy"
Critical

The root layout.tsx sets title.template = '%s | Bloggy'. Every child page that exports a plain string title (e.g., title: 'Sign In — Bloggy') has | Bloggy appended automatically, producing duplicates in the <title> tag.

Browser tab example: "Sign In — Bloggy | Bloggy"  ·  "All Tools — Bloggy | Bloggy"  ·  "Blog — AI Content Strategy & Agency Growth | Bloggy | Bloggy"

Google sees these exact strings. Duplicate branding in the title tag wastes characters, looks unprofessional in SERPs, and can trigger deduplication where Google rewrites your title.

Affected public pages: /, /features, /blog, /support, /signup, /terms, /privacy  ·  App pages (noindex, lower impact): /dashboard, /clients, /keywords, /autoblog, /calendar, /analytics, /settings, /billing, /tools, /help, /login

Fix — two clean options

Option A (recommended): Strip "— Bloggy" / "| Bloggy" from every page-level title string. The template appends it automatically. Result: 'Sign In' → "Sign In | Bloggy" ✓

Option B: Wrap titles in { absolute: '...' } to bypass the template where you need full control: title: { absolute: 'Bloggy — AI Blog Automation for Agencies' }

For the homepage only: Use absolute so "Bloggy" leads the title rather than becoming "Bloggy — ... | Bloggy".

C2 — /posts page: no metadata, no noindex — protected app page is indexable
Critical

The src/app/posts/page.tsx file exports no metadata object at all. Because it has no robots: { index: false }, Google can crawl it, but it will inherit only the root layout's generic title and description — meaning two pages share the same title tag in SERPs. The page requires authentication; Google will see a redirect or empty state.

Fix

Add a metadata export to posts/page.tsx: export const metadata = { title: 'Posts', description: '...', robots: { index: false, follow: false } }

C3 — No sitemap.xml — Google cannot efficiently discover all pages
Critical

There is no src/app/sitemap.ts and no public/sitemap.xml. Without a sitemap, Google relies entirely on links to find pages. Your 7 blog posts, /support, /features, and /signup are at risk of slow or incomplete indexing. A sitemap is especially important for a new site (bloggy.online is recently launched).

Fix — create src/app/sitemap.ts

Next.js generates /sitemap.xml automatically from this file. Include: /, /features, /blog, /blog/[each slug], /support, /signup, /terms, /privacy. Exclude all app pages. Submit URL to Google Search Console after deploy.

C4 — No robots.txt — no crawl guidance for Google and Bing
Critical

Neither public/robots.txt nor src/app/robots.ts exists. Without it: (1) no sitemap pointer for crawlers, (2) no explicit disallow for app routes like /dashboard, /api, /posts, (3) no block on crawl budget waste from protected routes that redirect to login.

Fix — create src/app/robots.ts

Allow all public paths, disallow /dashboard, /clients, /posts, /keywords, /autoblog, /calendar, /analytics, /settings, /billing, /api. Point Sitemap to https://bloggy.online/sitemap.xml.

High Priority Issues

H1 — /blog index: no structured data (missing ItemList / BlogPosting schema)
High

The blog listing page has good metadata but zero JSON-LD schema. Adding ItemList or Blog schema with each post listed as a BlogPosting item enables Google to show your posts as rich results in search — especially useful since the blog targets competitive agency SEO keywords.

Fix

In src/app/blog/page.tsx, add a <script type="application/ld+json"> block with @type: "Blog" containing a blogPost array of BlogPosting items (title, url, datePublished, description) using the BLOG_POSTS constant already imported on the page.

H2 — /support: no FAQPage JSON-LD — SupportFaq is a client component (no static schema)
High

SupportFaq.tsx is a 'use client' component — its content is rendered by JavaScript in the browser. Google's crawler does not reliably extract FAQ structured data from client-rendered components. The FAQPage JSON-LD must be in the server-rendered page.tsx for Google to use it in rich results.

This page has 20 FAQ questions across 5 categories — that is exactly the content Google wants for featured FAQ snippets. Right now none of it is marked up.

Fix

Add a <script type="application/ld+json"> block directly in src/app/support/page.tsx (server component) with @type: "FAQPage" and the top 8–10 FAQ Q&As hardcoded. The homepage already does this correctly with its FAQS constant — replicate the same pattern here.

H3 — /features: no page-level structured data
High

The features page ranks for "AI blog generator features" and "WordPress publish automation" — competitive product terms. No schema means no chance of a rich result. A SoftwareApplication or Product schema with your features listed as highlights would strengthen this page's SERP appearance.

Fix

Add ItemList schema in src/app/features/page.tsx listing key features as ListItem entries, or extend the global SoftwareApplication schema to include a featureList property.

H4 — Blog post pages: no keywords meta tag on any post
High

The generateMetadata function in blog/[slug]/page.tsx returns title, description, openGraph, twitter, and alternates — but no keywords array. While Google officially ignores the keywords meta tag, Bing still uses it, and it signals intent. More importantly, each post targets a specific focus keyword that should appear in the meta.

Fix

Add a keywords field to the BlogPost type in posts.ts, populate it per post (3–5 keywords), and include keywords: post.keywords in generateMetadata.

H5 — /terms and /privacy: no OpenGraph tags — blank social previews when shared
High

Both legal pages are indexable (robots: index) but have no openGraph data. If someone shares a link to your privacy policy on social media, Twitter/LinkedIn will show a blank card. These pages are also linked from every email footer and confirmation screen — they get shared.

Fix

Add minimal openGraph to both pages: { title, description, url, type: 'website', images: [og-image] }. Same pattern as /blog or /features.

H6 — /signup is fully indexable — no noindex, no canonical strategy defined
High

The signup page has metadata, OG tags, and a canonical URL — suggesting it is intentionally indexable, which is valid (conversion pages can rank). But it has no explicit robots directive, so it defaults to whatever Google decides. If your intent is to rank it for "Bloggy free trial" type searches, keep it indexable but confirm the decision. If not, add noindex.

Decision needed

If indexable (recommended for trial signups): add robots: { index: true, follow: true } explicitly. If not: add robots: { index: false, follow: false } to signup/layout.tsx.

H7 — No Twitter card on blog post pages (falls back to root layout summary_large_image)
High

The blog post generateMetadata returns a twitter object with card, title, description, and images — but it does NOT include a site handle (@GetBloggy). Without the handle, Twitter attribution is broken and the card may appear without brand linkage on X.

Fix (1 line)

Add site: '@GetBloggy' to the twitter object returned by generateMetadata in blog/[slug]/page.tsx.

Warnings (fix in next sprint)

W1 — Meta description length violations: /features (184 chars) and /support (177 chars) exceed 160-char limit
Warning

Google truncates descriptions around 155–160 characters in SERPs. Text after that is cut with "…" and the call-to-action at the end of both descriptions is invisible in search results.

/features: "Every Bloggy feature: AI blog generation from topics, URLs & YouTube, one-click WordPress publish, AutoBlog scheduler, keyword research, social repurposing, rankings tracker, and more." (184 chars)

/support: "Step-by-step guides, tutorials, and FAQs for Bloggy. Learn how to set up WordPress publishing, configure AutoBlog, track keyword rankings, and get the most from every feature." (177 chars)

Fix

Trim to ≤155 chars. Suggested for /features: "AI blog generation from topics, URLs & YouTube · one-click WordPress publish · AutoBlog scheduler · keyword research · social repurposing · rankings tracker." (154 chars)

W2 — Single generic OG image used on all pages (og-image.png)
Warning

Every page — homepage, features, blog index, blog posts, support, signup — uses the same /og-image.png (1200×630). When multiple pages are shared from the same domain, social platforms may show the same image repeatedly, reducing click differentiation. Blog posts especially benefit from unique preview images since each post already stores its own image URL in post.image.

Fix (blog posts already have post.image)

Blog posts already have post.image from the Unsplash URLs in posts.ts. The generateMetadata in blog/[slug]/page.tsx already uses post.image for OG — that part is correct. For other pages, consider creating 3–4 page-specific OG images for /features, /support, /signup for better social CTR.

W3 — Google Search Console not confirmed — site may not be verified
Warning

GA4 is wired up (G-WT6X97C78G) and Google Ads conversion tracking is present (AW-18109945240), but there is no Search Console verification meta tag or DNS TXT record visible in the codebase. Without GSC: no index coverage data, no manual indexing requests, no Core Web Vitals from field data, and no keyword performance in Search Console.

Action needed (not a code change)

Verify at search.google.com/search-console using the DNS TXT method (preferred) or by adding the Google Site Verification meta tag to layout.tsx metadata. Submit sitemap.xml once created.

W4 — No canonical URL set on /terms and /privacy OpenGraph
Warning

Both pages have alternates: { canonical: '...' } set correctly. But both are missing an openGraph.url which serves as the OG canonical. If shared, the og:url may default to the sharing URL (with query params or UTM) rather than the clean canonical.

Fix

Add openGraph: { url: 'https://bloggy.online/terms' } to both pages when you add OG tags (fixes H5 and W4 together).

W5 — Blog post pages lack a Twitter site handle
Warning

Covered in H7 above. site: '@GetBloggy' missing from blog post Twitter card. One-line fix.

W6 — No page-level H1 audit possible — confirm H1 exists and matches title intent on all public pages
Warning

Static analysis only. Recommend manually verifying that each public page has exactly one <h1> tag, that it contains the primary focus keyword, and that it matches the semantic intent of the title tag. Common issues: missing H1 on marketing pages, H1 hidden in a client component, H1 not matching the meta title.

Pages to manually check: / (homepage), /features, /blog, /blog/[slug], /support

Schema Rating Note

The 3 global schemas (SoftwareApplication, Organization, WebSite) injected in layout.tsx are on every single page — including noindex app pages. While harmless, best practice is to put global schema only on pages that benefit (homepage, landing pages). Google may also see multiple SoftwareApplication schema from the same URL across different pages as redundant. Consider moving these to the homepage page.tsx instead of the root layout.

What's Working Well

All protected app pages correctly marked noindex
Pass

Dashboard, clients, keywords, autoblog, calendar, analytics, settings, billing, tools, help, and login all have robots: { index: false, follow: false }. Google will not index any authenticated app page. Exception: /posts (C2 above).

Article JSON-LD on all 7 blog posts — correct server-side injection
Pass

Every blog post injects an Article schema via <script type="application/ld+json"> directly in the server component. Schema includes headline, description, image, datePublished, dateModified, url, author (Organization), publisher (with logo). This is exactly what Google requires for Article rich results.

FAQPage schema on homepage — hardcoded in server component
Pass

The homepage has 9 FAQ Q&As in a FAQPage JSON-LD block injected server-side. This is the correct approach and qualifies for Google's FAQ rich snippets in search results. The /support page should mirror this approach for its 20 FAQs (see H2).

3 global schemas in root layout: SoftwareApplication, Organization, WebSite
Pass

The SoftwareApplication schema includes aggregateRating (4.9 stars, 47 reviews) and AggregateOffer ($49–$299). Organization schema includes contactPoint and sameAs social URLs. WebSite schema has SearchAction for sitelinks searchbox. All structurally valid.

OpenGraph and Twitter cards on all key public pages
Pass

Homepage, features, blog index, blog posts, support, and signup all have proper OG metadata (title, description, url, type, image 1200×630). Blog posts have OG with publishedTime and authors for article type. Twitter summary_large_image with @GetBloggy handle on root level.

Canonical URLs on all public pages
Pass

Every public page sets alternates: { canonical: 'https://bloggy.online/...' }. Blog posts use dynamic canonical per slug. This prevents duplicate content issues from Vercel preview URLs, query parameters, or trailing slashes.

GA4 + Google Ads conversion tracking active
Pass

GA4 (G-WT6X97C78G) fires on every page via Next.js Script with strategy="afterInteractive". Google Ads (AW-18109945240) also fires for conversion tracking. Both use the non-blocking afterInteractive strategy — no page speed impact.

Blog posts statically generated (generateStaticParams)
Pass

All 7 blog post pages use generateStaticParams() — they are built as static HTML at deploy time. Google gets instant, fully-rendered HTML with no JavaScript required for content. Zero crawl budget wasted on JS rendering. Core Web Vitals will be excellent for blog pages.
