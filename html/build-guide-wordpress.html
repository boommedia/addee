Overview

How WordPress Publishing Works

WordPress has a built-in REST API. We authenticate with an "Application Password" (a special secure token) and POST the blog content as HTML directly to the client's site.

What we need per client
3 things from WordPress

Site URL · WordPress username · Application Password (not their login password)

WordPress requirement
WordPress 5.6+ (2020)

Application Passwords are built-in from WP 5.6. Works on any self-hosted WordPress. Does NOT work on WordPress.com free plans.

Published post status
Draft by default

We publish as "draft" so client can review before going live. One setting change to publish immediately if preferred.

WordPress needs HTML. Claude returns markdown. The marked package converts it instantly.

Phase 1 — On the Client's WordPress Site

WordPress Setup (Done Once Per Client)

You or the client needs to do this on each WordPress site. Takes 3 minutes. Only needs to be done once per site.

1

Enable Application Passwords on the WordPress Site

On Client's Site
ℹ️

Application Passwords are different from the login password. They are special tokens generated for API access only. If one is compromised, you delete it and generate a new one — the actual account password is untouched.

Step 1 — Log into the client's WordPress admin

Go to https://clientsite.com/wp-admin and log in as an Administrator.

Step 2 — Check Application Passwords are not blocked

Some security plugins (Wordfence, iThemes Security) disable the REST API or Application Passwords. To verify the REST API is accessible, open a browser tab and go to:

https://clientsite.com/wp-json/wp/v2/posts

If you see a JSON response (even if it's an empty array []), the REST API is working. If you get an error or a blank page, the REST API is blocked — see the callout below.

⚠️

If the REST API is blocked: In WordPress admin go to the security plugin settings (Wordfence → Firewall, or iThemes → Security → Settings) and look for "Disable REST API" or "REST API access" — set it to allow authenticated users or disable the block entirely. If using Wordfence: Wordfence → Firewall → Brute Force Protection → uncheck "Disable WordPress XML-RPC authentication" (different feature but sometimes confused).

Step 3 — Generate an Application Password

  • In WordPress admin: go to Users → Profile (or Users → All Users → click the admin user → Edit)
  • Scroll all the way down to the "Application Passwords" section
  • In the "New Application Password Name" field, type: Boom Media Blog Tool
  • Click "Add New Application Password"
  • WordPress shows the password one time only — it looks like: AbCd EfGh IjKl MnOp QrSt UvWx
  • Copy it immediately. Remove the spaces when saving — store as: AbCdEfGhIjKlMnOpQrStUvWx

What to record for each client

# Save these 3 values for each client in the Boom Media tool:

WordPress Site URL:   https://clientsite.com
WordPress Username:   admin              (the username they log in with)
Application Password: AbCdEfGhIjKlMnOpQrStUvWx  (no spaces)
✓

Use an Editor or Author account, not Admin: For extra security, create a dedicated WordPress user with "Editor" role for API publishing. That way the API token can only create/edit posts — not change site settings. Name the user boom-media-api.

Security note: Application passwords are stored as plain text in Supabase for this internal v1. Since this is a private internal tool with controlled access, this is acceptable. For a future public SaaS version, encrypt these with AES-256 before storing using a separate encryption key.

Phase 4 — Backend

Create the Publish API Route

A new server-side route that handles the publish request. WordPress credentials never touch the browser.

5

Create app/api/publish-wordpress/route.ts

New File

Create this file at the path shown in the label — make the folder app/api/publish-wordpress/ first if it doesn't exist.

app/api/publish-wordpress/route.ts
import { NextRequest, NextResponse } from 'next/server'
import { marked } from 'marked'
import { createClient } from '@/lib/supabase/server'

export async function POST(request: NextRequest) {
  try {
    const { postId, publishStatus = 'draft' } = await request.json()

    if (!postId) {
      return NextResponse.json({ error: 'postId is required' }, { status: 400 })
    }

    const supabase = await createClient()

    // Verify the user is logged in
    const { data: { user } } = await supabase.auth.getUser()
    if (!user) {
      return NextResponse.json({ error: 'Unauthorized' }, { status: 401 })
    }

    // Load the blog post and its client's WordPress credentials
    const { data: post, error: postError } = await supabase
      .from('blog_posts')
      .select(`
        id,
        title,
        content,
        meta_description,
        client_id,
        clients (
          name,
          wp_url,
          wp_username,
          wp_app_password
        )
      `)
      .eq('id', postId)
      .single()

    if (postError || !post) {
      return NextResponse.json({ error: 'Blog post not found' }, { status: 404 })
    }

    const client = (post as any).clients

    // Validate WordPress credentials exist for this client
    if (!client?.wp_url || !client?.wp_username || !client?.wp_app_password) {
      return NextResponse.json({
        error: 'WordPress credentials not configured for this client. Go to Clients → Edit and add the WordPress URL, username, and application password.',
      }, { status: 422 })
    }

    // Convert Markdown content to HTML for WordPress
    const contentHtml = await marked(post.content || '', {
      gfm: true,     // GitHub Flavored Markdown (tables, strikethrough, etc.)
      breaks: true,  // Convert \n to 
 in paragraphs
    })

    // Build the WordPress post payload
    const wpPayload = {
      title: post.title,
      content: contentHtml,
      status: publishStatus, // 'draft' or 'publish'
      excerpt: post.meta_description || '',
    }

    // Clean up the WordPress URL (remove trailing slash)
    const wpBaseUrl = client.wp_url.replace(/\/$/, '')
    const wpApiUrl = `${wpBaseUrl}/wp-json/wp/v2/posts`

    // Build Basic Auth header from username + application password
    const credentials = Buffer.from(
      `${client.wp_username}:${client.wp_app_password}`
    ).toString('base64')

    // POST to WordPress REST API
    const wpResponse = await fetch(wpApiUrl, {
      method: 'POST',
      headers: {
        'Content-Type': 'application/json',
        'Authorization': `Basic ${credentials}`,
      },
      body: JSON.stringify(wpPayload),
    })

    const wpData = await wpResponse.json()

    // Handle WordPress errors
    if (!wpResponse.ok) {
      console.error('WordPress API error:', wpData)

      const wpError = wpData?.message || wpData?.code || 'WordPress rejected the request'
      return NextResponse.json({
        error: `WordPress error: ${wpError}`,
        wpCode: wpData?.code,
      }, { status: wpResponse.status })
    }

    // WordPress published successfully — save the WP post details to our DB
    const { error: updateError } = await supabase
      .from('blog_posts')
      .update({
        wp_post_id: wpData.id,
        wp_post_url: wpData.link,
        wp_status: publishStatus,
        status: publishStatus === 'publish' ? 'published' : 'draft',
      })
      .eq('id', postId)

    if (updateError) {
      console.error('DB update error after WP publish:', updateError)
      // Don't fail the request — post IS on WordPress, just DB tracking failed
    }

    return NextResponse.json({
      success: true,
      wpPostId: wpData.id,
      wpPostUrl: wpData.link,
      wpStatus: publishStatus,
      wpEditUrl: `${wpBaseUrl}/wp-admin/post.php?post=${wpData.id}&action=edit`,
    })

  } catch (error: any) {
    console.error('Publish to WordPress error:', error)
    return NextResponse.json(
      { error: error.message || 'Failed to publish to WordPress' },
      { status: 500 }
    )
  }
}
✓

Two publish modes: The route accepts publishStatus: "draft" or publishStatus: "publish". The UI will offer both buttons. Draft = goes to WordPress but stays hidden. Publish = goes live immediately. Default is draft for safety.

async function handlePublishToWordPress(publishStatus: 'draft' | 'publish') {
  if (!output?.id) return
  setPublishing(true)
  setPublishError('')
  setPublishResult(null)

  try {
    const res = await fetch('/api/publish-wordpress', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ postId: output.id, publishStatus }),
    })

    const data = await res.json()

    if (!res.ok) {
      throw new Error(data.error || 'Publish failed')
    }

    setPublishResult({
      wpPostUrl: data.wpPostUrl,
      wpEditUrl: data.wpEditUrl,
      wpStatus: data.wpStatus,
    })

    // Update the local output state so the UI reflects the new WP status
    setOutput(prev => prev ? {
      ...prev,
      wp_post_id: data.wpPostId,
      wp_post_url: data.wpPostUrl,
      wp_status: data.wpStatus,
      status: publishStatus === 'publish' ? 'published' : 'draft',
    } : null)

  } catch (err: any) {
    setPublishError(err.message)
  } finally {
    setPublishing(false)
  }
}

Troubleshooting Common Issues

★

Common WordPress Errors and Fixes

Quick Reference

401 Unauthorized

WordPress error: Sorry, you are not allowed to do this.

The username or application password is wrong. Double-check: the username is the WP login username, not the display name or email. The application password should have no spaces. Re-generate a new application password in WP Admin → Users → Profile and try again.

403 Forbidden — REST API disabled

WordPress error: rest_forbidden

A security plugin is blocking the REST API. Common culprits: Wordfence, Solid Security (iThemes), All In One WP Security. Look for a setting called "Disable REST API" and set it to "Allow authenticated users" or disable it for the posts endpoint.

Cannot POST — fetch failed / ECONNREFUSED

The WordPress URL is wrong or the site is unreachable. Verify the URL is correct and the site is live. Check there's no redirect from http → https causing issues (always use the final https:// URL).

Posts publishing with broken formatting

You see raw markdown in WordPress (e.g. ## Heading instead of an H2). This means marked isn't running. Check: is npm install marked complete? Is the import at the top of the route file correct? Check the terminal for any TypeScript errors when the route runs.

Post published but Supabase not updating

The post IS on WordPress, but wp_post_id isn't saved in Supabase. This is a non-critical error — the API still returns success. Check Supabase logs (Dashboard → Logs) for the RLS policy blocking the update. The fix: make sure the Supabase client in server.ts uses the anon key (not service role) since the user's session handles auth.

Image in blog content not appearing on WordPress

Expected — v1 generates text-only posts. Claude doesn't produce images yet (that's a future feature). WordPress will show the post without a featured image. The client's WP theme will handle layout without it.
